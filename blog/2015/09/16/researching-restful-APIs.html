Researching RESTful APIs

Posted on: September 16, 2015

It has been a while since I've updated my site. Mainly because I have no time. I'm always working whether it be from my retail job on the weekends or my city job on the weekdays right before school. Thankfully that crazy schedule is coming to an end. Today I gave in my two-weeks notice to my retail job. Anyways that is besides the point of why I'm here. Today I decided to look up RESTful APIs. First the simple stuff and API. From what I know API means - Application Program Interface which are a set of routines, protocols, and tools for building software applicatons. Kind of like the messenger that takes requests, tells the system what you want to do and then gives the response back to you. Example would be an searching for airline flights online. Think of the websites dropdown menu as an actual menu at a restaurant. Pick and choose departure dates, departure cities and so on. In order to get a response you must access the website's database. The same can be said about other a travel service that aggregates information from many different airlines. The travel service interacts with the airline's API. The API like the waiter can be asked by the travel service to request information from the airline's system to book seats, meal options or baggage options. It then takes the airline's response to your request and sends it back to the travel service to be presented in a more comprehendable way. REST API is something totally new to me so I had to look it up in order to understand it.REST stands for REpresentational State Transfer - and it relies on stateless, client-server, cacheable communications protocol -- and virtually all cases, the HTTP protocol is used. REST is an architecture style for designing networked application. It works most similarly like a website. One makes calls from a client to a server and you receive data back via the HTTP protocol.
